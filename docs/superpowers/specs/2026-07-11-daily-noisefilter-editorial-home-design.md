# 降噪每日简报与首页改版 Design Spec

> 状态：用户已确认最终设计方向
>
> 日期：2026-07-11
>
> 仓库：`content-curation`
>
> 本文只定义产品、交互、数据与验收标准，不包含实现步骤或代码修改。

## 1. 一句话定位

> 给 AI 从业者的每日降噪简报：自己快速看懂，也能直接转给同事。

系统把围绕同一事件的重复消息合并成一个讨论主题，每天留下最多三个值得知道的主题。个人用户先用它快速学习，觉得重要时可把同一期固定链接直接分享给公司同事。

本产品当前不是公司内部情报系统，也不假装了解某家公司的具体业务。未来有真实公司场景后，再增加非公开的“与本公司业务的关系”。

## 2. 本轮目标

1. 把首页从“实时 AI HOT/判断面板”改造成清晰的每日编辑简报。
2. 让普通读者打开首页后，在约 10 秒内知道今天最值得关注的 1、2、3。
3. 将实时信号与已发布简报分离，确保分享出去的历史页面不会漂移。
4. 保留人工最终判断：系统预选、合并并排序，管理员发布前可以调整。
5. 保留“热议浮现”和现有深度内容库，但把它们放在今日前三之后。

## 3. 已确认的产品边界

### 3.1 本轮包含

- 每日一期、按日期固定保存。
- 首页始终展示最新已发布的一期。
- 系统自动预排候选，管理员发布前可排序、移除和替换。
- 管理员手动发布；不做定时自动发布。
- 同一期允许修订，链接不变，并显示最后修订时间。
- 今日可少于三条，宁缺毋滥。
- 事件级去重与相关来源合并。
- 公开页面展示轻量的“为什么上榜”和真实来源依据。
- 一键复制本期，便于分享给公司同事。

### 3.2 本轮不包含

- “通用版 / 技术版”等阅读模式切换。
- 两套独立选题或两套简报。
- 公司内部业务分析、公司知识库接入或敏感备注。
- 邮件、RSS、微信群或飞书群自动推送。
- 多人协作编辑、审批流或团队权限系统。
- 自动定时发布。
- 深度内容库卡片与图片区域的重设计。
- 报刊/报纸式视觉方案。
- 对外展示算法分数、热度仪表盘或 `AI HOT` 字样。

## 4. 与现有系统的关系

现有链路继续负责生成实时候选：

```text
AI HOT / RSS / 外部来源 / archive
              ↓
     today_signal.json（实时草稿输入）
              ↓
      事件合并、预排、人工编辑
              ↓
 daily_issues/YYYY-MM-DD.json（已发布快照）
              ↓
      首页 / 日期页 / 团队分享链接
```

关键边界：

- `blog/data/today_signal.json` 是会继续变化的草稿输入，不再直接代表公开首页。
- 已发布简报是独立快照，不随实时缓存、重建索引或新内容入库自动变化。
- 只有管理员明确发布或修订，公开简报才会改变。
- 旧版 `今日 AI 判断可信度与学习闭环` spec 中的信号生成、新鲜度、来源核验等底层能力继续有效；本文取代其公开首页布局、`AI HOT` 展示和实时缓存直接上首页的产品语义。

## 5. 核心内容单位：事件主题，不是文章

首页排名对象是“事件主题”。围绕同一发布、产品变化或政策事件的多篇文章、转述和讨论，必须先合并，再决定是否进入前三。

示例：

```text
官方发布 + 多家媒体报道 + 多条社交讨论
                    ↓
           “某重大模型发布”一个主题
```

不能出现以下情况：

- 同一发布因标题不同占据两个排名。
- 将转载数量直接当作独立信源数量。
- 为了让页面显得可信而编造“12 个来源”等数字。
- 只有热度，没有明确事实或可靠依据的主题直接进入前三。

独立来源计数规则：

- URL 完全相同的内容只计一次。
- 同一来源的重复转载默认只计一次。
- 官方原始发布与独立媒体/研究者分析分开计数。
- 页面仅在真实独立来源不少于两个时显示“X 个相关来源已合并”。
- 未达到条件时只显示来源名称，不显示合并数量。

## 6. 排序原则

排序不是纯热度榜。内部按以下顺序形成候选：

1. **讨论广度负责发现**：识别多个独立来源共同关注的事件。
2. **新闻重要性负责纠偏**：重大模型发布、平台政策变化、关键产品停用等事件，即使讨论刚开始，也可以进入高位。
3. **信源质量负责兜底**：优先官方原始来源、可信媒体、研究机构和可核验材料。
4. **时效与新颖性防止旧闻回流**：重复旧观点不能仅凭再传播进入前三。
5. **管理员最终确认**：系统排序只是草稿，最终顺序代表编辑判断。

公开页面不展示计算权重、分数或仪表盘，只展示：

- 相关来源已合并的真实数量（符合显示条件时）。
- 一句“为什么上榜”。
- 可展开的主要依据。

管理员对候选的拖动、替换、移除和发布结果应被记录，作为后续改进排序的反馈，但第一版不自动训练复杂模型。

## 7. 页面信息架构

### 7.1 顶部导航

普通读者可见：

- 已确认的“降噪”Logo 与暖色背景。
- 往期简报。
- 深度内容库。
- 搜索。
- `复制本期`。

管理员额外可见：

- 当天未发布：`发布今日`。
- 当天已发布：`修订本期`。

普通读者永远看不到发布、排序、替换、加入首页、先不加或加入深度库等管理按钮。

### 7.2 简报题头

首页最新一期：

```text
2026.07.11 · 第 027 期
今天，哪些讨论值得知道？
不是热搜榜。把重复消息合并后，只留下可能影响下一步判断的变化。
今天留下 3 个主题 · 不凑数
```

历史日期页将“今天”改为“这一天”，避免读者打开旧链接时产生时间误解：

```text
2026.07.10 · 第 026 期
这一天，哪些讨论值得知道？
```

### 7.3 今日前三

桌面端采用已确认的卡片布局：

```text
┌──────────────────────────────┬──────────────────┐
│                              │  2 · 小卡片      │
│       1 · 左侧大卡片         ├──────────────────┤
│                              │  3 · 小卡片      │
└──────────────────────────────┴──────────────────┘
```

视觉规则：

- 暖米色页面背景。
- 第 1 条使用左侧大圆角卡片和轻暖色底。
- 第 2、3 条使用右侧上下排列的小圆角卡片。
- 蓝色用于来源/状态等辅助信息，红色只做轻量焦点与 Logo 端点。
- 主要依靠字号、空间和卡片尺寸表达优先级，不依靠大面积深色或复杂指标。
- 不采用报刊式黑色细线排版，也不采用 SaaS 大 Hero。

卡片内容：

| 字段 | 第 1 条大卡片 | 第 2、3 条小卡片 |
|---|---|---|
| 排名 | 必须 | 必须 |
| 类别 | 可显示 | 可显示 |
| 相关来源已合并 | 符合真实计数规则时显示 | 符合时显示 |
| 标题 | 必须 | 必须 |
| 发生了什么 | 1–2 句 | 1 句 |
| 大家主要在讨论 | 最多 3 个短焦点 | 默认不展开 |
| 为什么上榜 / 降噪判断 | 必须 | 必须，短句 |
| 查看依据 | 必须 | 可点击详情后查看 |

内容长度约束：

- 第 1 条标题建议不超过 34 个中文字符。
- 第 2、3 条标题建议不超过 24 个中文字符。
- “发生了什么”只陈述一个核心事实，不复述完整新闻。
- “大家主要在讨论”最多三个短语。
- “为什么上榜”只解释重要性，不重复摘要。
- 发布预览对过长内容给出提示；公开页面使用合理的行数约束防止布局失控。

### 7.4 条目不足三条

- 三条：标准“左一大、右二小”布局。
- 两条：左侧大卡片 + 右侧一张小卡片，右侧卡片垂直居中或自然顶部对齐，不渲染空白占位卡。
- 一条：第 1 条扩展为整行主卡片。
- 零条：不允许发布空简报。

页面文案明确写“今天仅有 N 个主题达到标准”，不使用旧内容补齐。

### 7.5 热议浮现

位于今日前三之后，需向下滚动才能完整看到。

语义：

> 正在升温，但目前还不足以进入今日前三。

每条只显示：

- 主题标题。
- 一句当前状态，例如“等待更多独立信源”“真实采用证据仍然有限”。
- 轻量状态：讨论上升中 / 继续观察 / 讨论稳定。

普通读者不能在这里执行“加入首页判断”或“先不加”。这些操作只属于管理员发布流程。

### 7.6 深度内容库

- 保持现有图片、卡片结构与用户已经认可的视觉风格。
- 只调整它与“热议浮现”之间的间距和分区衔接。
- 不在本轮重写详情页或内容库信息架构。

## 8. 每日简报数据模型

公开快照保存为：

```text
blog/data/daily_issues/YYYY-MM-DD.json
blog/data/daily_issues/revisions/YYYY-MM-DD-rNN.json
```

`blog/data/` 已是本地生成数据且被忽略，不提交公开简报内容到 Git。

快照结构如下：

```json
{
  "schema_version": 1,
  "issue_date": "2026-07-11",
  "issue_number": 27,
  "status": "published",
  "revision": 2,
  "published_at": "2026-07-11T09:10:00-07:00",
  "revised_at": "2026-07-11T09:42:00-07:00",
  "timezone": "America/Los_Angeles",
  "topics": [
    {
      "topic_id": "stable-topic-id",
      "rank": 1,
      "category": "模型发布",
      "title": "示例标题",
      "what_happened": "一句核心事实",
      "discussion_focus": ["焦点一", "焦点二", "焦点三"],
      "why_ranked": "为什么值得今天先知道",
      "independent_source_count": 6,
      "sources": [
        {
          "title": "来源标题",
          "url": "https://example.com/source",
          "publisher": "来源名称",
          "is_primary": true
        }
      ]
    }
  ],
  "attention": []
}
```

约束：

- `topics` 长度只能是 1–3。
- `rank` 连续且从 1 开始。
- 同一期修订不改变 `issue_date`、`issue_number` 或公开 URL。
- 每次修订前先保存旧版本到 `revisions/`。
- 当前版本使用临时文件写入后原子替换，避免半写入状态。
- issue number 在没有历史简报时从 1 开始，仅在首次发布新日期时递增；修订不递增。
- 团队分享文本由当前快照字段动态生成，不另存冗余副本，避免修订后文案与页面不一致。

## 9. 路由与公开语义

### 9.1 路由

- `/`：读取并展示最新已发布一期。
- `/daily/<YYYY-MM-DD>`：读取指定日期的固定快照。
- `/daily`：往期列表，按日期倒序。
- 管理员发布与修订接口置于 `/admin/daily/...`，必须经过管理员身份校验。

首页可以复用日期页模板，但文案必须根据 issue date 判断：

- `/` 展示的 issue date 等于应用本地当天时使用“今天”。
- `/` 展示的是更早的最新一期时使用“最新一期 / 这一天”，不能暗示今天已经更新。
- 历史日期页使用“这一天”。
- 分享按钮永远复制带日期的稳定 URL，而不是 `/`。
- 日期页输出 canonical URL、标题、摘要和分享图元数据。

### 9.2 没有已发布简报

首次使用且没有任何已发布一期时：

- 首页显示“还没有发布第一期”。
- 管理员可进入草稿并发布。
- 普通读者仍可以继续浏览下方深度内容库。
- 不把实时草稿伪装成已发布简报。

### 9.3 最新一期不是今天

管理员某天未发布时：

- 首页继续展示最新一期，日期必须清晰可见。
- 文案使用“最新一期”，不伪装成今天已更新。
- 管理员按钮显示 `发布今日`；只有当前展示日期就是今天且已经发布时才显示 `修订本期`。
- 不自动用旧主题生成新日期页面。

## 10. 管理员发布与修订流程

### 10.1 草稿

系统从实时候选中自动生成最多三条预排主题，并准备额外候选列表。普通读者看不到草稿。

### 10.2 发布确认窗口

点击右上角 `发布今日` 后显示：

- 日期、期号与候选生成时间。
- 当前 1、2、3。
- 拖动排序。
- 移除主题。
- 从额外候选中替换。
- 在展开状态下编辑标题、核心事实、讨论焦点和“为什么上榜”。
- 每条的来源与“为什么上榜”。
- 内容长度和来源不足提示。
- `预览读者页面`。
- `确认发布`。

系统允许发布一条或两条，不允许发布零条。

### 10.3 修订

当天或历史日期页已发布时，管理员按钮变为 `修订本期`。

修订规则：

- 可以调整排序、文案、来源和入选主题。
- 保存前归档旧 revision。
- 成功后公开 URL 不变。
- 页面显示“最后修订 HH:MM”。
- 修订失败时继续提供上一成功版本。

### 10.4 管理员隔离

- 第一版使用单管理员密码和 Flask 签名 session，不建立用户表或多账号系统。
- 管理员密码由 `config/.env` 的 `BLOG_ADMIN_PASSWORD` 提供，绝不写入模板、URL、日志或 Git。
- 管理员通过 `/admin/login` 建立 session；外部部署时 session cookie 必须使用 `HttpOnly`、`SameSite=Lax` 和 HTTPS 下的 `Secure`。
- 未认证访问管理接口返回 401/403。
- 公开模板不渲染管理按钮和管理数据属性。
- 第一版只需单管理员，不扩展多角色权限模型。

## 11. 分享场景

`复制本期` 生成适合公司群聊的纯文本：

```text
降噪｜2026 年 7 月 11 日

今天值得知道的三个讨论：
① 标题 + 一句判断
② 标题 + 一句判断
③ 标题 + 一句判断

完整简报：https://example.com/daily/2026-07-11
```

规则：

- 只有一条或两条时按真实数量生成。
- 复制的是日期页稳定链接。
- 不自动发送外部消息，发送动作仍由用户完成。
- 日期页提供正确的 Open Graph 标题、摘要和分享图信息。

## 12. 异常与边界状态

| 情况 | 管理员 | 普通读者 |
|---|---|---|
| 今日信号尚未生成 | 发布按钮禁用，提示先生成信号 | 继续看到上一已发布期 |
| 候选少于三条 | 允许发布，明确提示数量 | 显示真实数量，不凑数 |
| 没有候选 | 禁止发布空简报 | 继续显示上一期 |
| 保存失败 | 显示错误并保留草稿 | 继续显示上一成功版本 |
| 同日期重复发布 | 自动进入修订流程 | 不产生重复日期页 |
| 来源链接失效 | 可替换来源或修订说明 | 保留已发布判断并标记来源暂不可访问 |
| JSON 损坏 | 阻止覆盖并提示恢复 | 回退上一可读 revision 或友好错误页 |
| 热议浮现为空 | 不渲染该区 | 直接衔接深度内容库 |
| 最新一期日期较旧 | 提示今天尚未发布 | 明确显示“最新一期”及真实日期 |

## 13. 响应式与可访问性

### 13.1 桌面端

- 目标视口：1366×768 时，题头和今日 1、2、3 均进入首屏。
- 第 1 条宽度约占内容区 60%–65%，第 2、3 条约占 35%–40%。
- 热议浮现允许只露出开头，鼓励自然下滑。

### 13.2 移动端

- 按 1 → 2 → 3 垂直排列。
- 不强求三条全部进入一个手机首屏，但必须让第 2 条露出或通过清晰的序号提示仍有后续。
- 管理员编辑流程优先保证可用，不要求移动端拖拽；可提供上移/下移按钮作为替代。

### 13.3 可访问性

- 排名、状态和重要性不能只依赖颜色表达。
- 管理按钮使用真实 `button`，来源使用真实链接。
- 键盘可完成排序替代操作、发布确认与复制。
- 正文、辅助文字和暖色背景满足合理对比度。
- 尊重 `prefers-reduced-motion`，不使用无必要的大动画。

## 14. 测试与验收

### 14.1 数据与服务测试

- 事件去重不会让同一 URL 或同一来源重复计数。
- 公开来源数量与保存的数据一致，绝不使用示意数字。
- 新日期首次发布正确递增 issue number。
- 同日期再次保存产生新 revision，但 URL 和 issue number 不变。
- 原子写入失败时，上一版本仍可读取。
- `/` 正确选择最新已发布一期。
- `/daily/<date>` 不受 `today_signal.json` 后续变化影响。
- 1、2、3 条以及无 attention 的数据均可正常渲染。

### 14.2 权限测试

- 普通读者看不到管理员按钮。
- 未认证请求不能调用发布、修订或候选调整接口。
- 管理员凭证不会出现在 HTML、日志或分享文本中。

### 14.3 页面与交互测试

- 1366×768 下可看到完整的左一大、右二小布局。
- 长标题、长来源名和缺少类别时布局不溢出。
- 一条和两条简报没有空白占位卡。
- 复制分享文本使用稳定日期链接。
- 历史页使用“这一天”，首页最新当天使用“今天”。
- 热议浮现为空时与深度内容库自然衔接。
- 现有深度内容库的图片、卡片链接和详情入口没有回归。

### 14.4 产品验收标准

- 新读者约 10 秒内可以复述今天排名前三的主题。
- 页面不出现 `AI HOT`、受众模式切换或复杂评分仪表盘。
- 同一事件只出现一次。
- 排名既能反映讨论广度，也允许重大新闻通过重要性进入高位。
- 已分享的日期链接第二天打开仍显示原期内容。
- 管理员可以在发布前完成预排、调整和确认；发布失败不会影响公开首页。

## 15. 最终设计决定

- 产品切入点：个人先读，顺手分享给同事。
- 编辑单位：事件主题，不是文章。
- 标题：`今天，哪些讨论值得知道？`
- 首屏：第 1 条左侧大卡片，第 2、3 条右侧两个小卡片。
- 背景：已确认的暖米色。
- 信息顺序：`今日前三 → 热议浮现 → 深度内容库`。
- 排序：热度发现、重要性纠偏、信源质量兜底、人工最终确认。
- 发布：管理员手动发布；同一期允许修订。
- 历史：按日期固定，首页永远展示最新一期。
- 视觉明确排除：报刊版、双阅读模式、SaaS Hero、公开算法面板。
